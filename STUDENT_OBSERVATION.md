# AI 피지컬 컴퓨팅 — 차시별 학생 관찰 기록 (S~E)

> 지도서 방향 정렬을 위한 참고 문서
> 생활기록부 세부능력 및 특기사항 형식으로 작성
>
> **등급 기준**
> - S: 뛰어나고 놀라운 학생. 한 학교에 한두 명 있을까 말까 한 수준. 다른 교과로의 전이, 동료 지원이 자연스러움.
> - A: 기대 이상의 학생. 교재 범위를 넘는 자발적 탐구, 독창적 분석.
> - B: 이상적 학생. 핵심 역량을 체득하고 자기 언어로 표현. 최고의 배움.
> - C: 어느 정도 이상적인 학생. 좋은 배움.
> - D: 중간 정도 학생. 의미 있는 배움.
> - E: 조금 부족하지만 최소한의 배움은 달성.

## 1막: 센서와 친해지기 (직접 코딩)

---

### 1차시: 센서로 세상 읽기 (온습도 센서 DHT20)

**S** DHT20 센서로 교실 5개 지점의 온도를 측정한 뒤, 에어컨 토출구와의 거리를 변수로 놓고 온도 분포를 좌표 위에 그려 열전달 패턴을 시각화함. 측정 간격을 0.5초로 줄이는 실험을 자발적으로 수행하며 응답 속도를 탐구하는 한편, 옆자리 친구가 I2C 포트를 헷갈려 하자 "커넥터 모양이 같은 쪽에 꽂으면 돼"라고 자연스럽게 도와줌. 수업 후 과학 시간에 기온 측정 실험을 하면서 "우리가 피지컬 컴퓨팅 시간에 쓴 DHT20이면 소수점 한 자리까지 정확하게 나올 텐데"라며 센서 정밀도를 과학 실험 맥락에 연결하는 전이를 보이고, 쉬는 시간에 교실 온도 지도를 그려 게시판에 붙여 학급 친구들의 자리 선호도 논의를 촉발함.

**A** DHT20 센서를 연결하고 교실 내 5개 지점의 온도를 측정한 뒤, 에어컨 토출구와의 거리에 따른 온도 분포를 좌표 위에 그려 열전달 패턴을 시각화함. "체감 온도와 실측 온도의 오차가 왜 생기는지"를 습도와 기류 변수까지 연결하여 설명하고, 팀원들에게 측정 위치 분담을 제안하여 데이터 수집 효율을 높임. 측정 간격을 0.5초로 줄여 온도 변화의 응답 속도를 실험하는 등 교재 범위를 넘는 탐구를 자발적으로 수행함.

**B** DHT20 센서를 I2C 포트에 정확히 연결하고, MicroPython 코드를 직접 입력하여 교실 내 4~5개 지점의 온도를 측정함. 에어컨 근처(22.3도)와 창가(26.1도)의 차이를 발견하고, "같은 교실인데 자리마다 체감이 다른 이유가 센서 데이터로 증명된다"고 표현하며 사람의 감각과 센서의 객관적 수치를 비교함. 코드에서 `time.sleep(2)` 값을 1로 바꿔보고 측정 주기가 달라지는 것을 직접 관찰하는 등 코드의 각 줄이 실제 동작에 미치는 영향을 탐구함. 측정 결과를 표로 정리하여 팀원들과 공유하고, 위치별 온도 차이의 원인을 논의함.

**C** 센서를 올바른 포트에 연결하고 예제 코드를 입력하여 온습도 값을 정상적으로 읽어냄. 교실 3개 지점 이상을 측정하고 위치별 온도 차이가 있음을 확인하며, "에어컨 쪽이 더 시원하다"는 결과를 데이터로 뒷받침함. 코드의 반복 구조(`while True`)와 출력 형식(`f-string`)의 역할을 이해하고, 출력되는 두 숫자가 각각 온도와 습도에 해당한다는 것을 명확히 구분함.

**D** 교사의 안내에 따라 센서를 연결하고 코드를 실행하여 온습도 값이 출력되는 것을 확인함. 교실 2개 지점을 측정하고 온도가 다르다는 사실은 인식하나, 그 원인에 대한 구체적 분석까지는 이르지 못함. 코드를 따라 입력할 수 있으나 각 줄의 역할 설명은 부분적임.

**E** 센서 연결과 코드 입력에 교사의 도움을 받아 수행함. 코드 실행 후 화면에 숫자가 나오는 것을 확인하고, 센서가 교실의 온도를 읽고 있다는 기본 개념은 이해함.

### 2차시: 빛을 숫자로 (빛 센서 + Thonny Plotter)

**S** ADC 값 65535가 16비트(2^16)에서 유래한다는 것을 스스로 알아내고, 수학 시간에 배운 거듭제곱과 연결하여 "2진수 16자리의 최댓값"이라고 설명함. Plotter 그래프에서 빛 변화의 기울기를 "수학에서 배운 변화율과 같다"고 연결하며, 교실 조도 측정 결과를 바탕으로 "창가 학생과 복도 쪽 학생의 조도 차이가 3배인데, 보건 기준으로는 괜찮은 건지"를 조사해 오겠다고 자발적으로 과제를 설정함. 코드 입력에 어려움을 겪는 친구에게 다가가 "이 줄은 센서 값을 읽는 부분이고, 이 줄은 백분율로 바꾸는 거야"라고 코드를 한 줄씩 설명해 주며, 가르치는 과정에서 자신의 이해도 더 깊어지는 모습을 보임.

**A** ADC 값 65535가 16비트(2^16)에서 유래한다는 것을 스스로 알아내고, 0~100% 변환 공식의 수학적 의미를 설명함. Plotter 그래프에서 빛 변화의 기울기를 관찰하며 "급격한 변화와 완만한 변화의 차이가 그래프의 경사도로 보인다"고 표현함. 교실 조도 데이터를 수집한 뒤 "창가 학생의 눈 피로도가 높을 수 있다"며 학습 환경 개선안을 자발적으로 제시하고, 시간대별 조도 변화를 추적하는 실험을 제안함.

**B** 빛 센서를 A0 포트에 연결하고 ADC 값을 백분율로 변환하는 코드를 작성함. Thonny Plotter에 실시간 그래프가 그려지자 손으로 가리거나 손전등을 비추며 그래프의 반응을 관찰하고, "숫자만 볼 때는 몰랐는데 그래프로 보니 변화의 속도가 보인다"며 시각화의 가치를 체감함. 교실 5개 지점의 밝기를 표로 정리하고, 창가와 복도 쪽 조도 차이가 학습 환경에 미치는 영향에 대해 자발적으로 의견을 제시함. Pico 내장 LED를 밝기에 연동시키는 코드를 시도하며 입력-출력 대응 구조를 탐구함.

**C** 센서를 연결하고 Plotter로 실시간 그래프를 확인하며, 손으로 센서를 가릴 때 그래프가 내려가고 손전등을 비추면 올라가는 인과관계를 명확히 이해함. 교실 3개 지점 이상의 밝기를 측정하고 위치별 차이를 기록하며, ADC 값을 백분율로 바꾸는 나눗셈 코드의 의미를 설명할 수 있음. 그래프의 x축이 시간, y축이 밝기를 나타낸다는 것을 파악하고 그래프 읽기에 자신감을 보임.

**D** 교사의 안내에 따라 센서를 연결하고 Plotter 기능을 사용함. 그래프가 움직이는 것을 보고 재미를 느끼며, 센서를 가릴 때 값이 줄어드는 것을 확인함. 교실 측정에 참여하여 2개 이상 지점의 값을 기록함.

**E** 센서 연결과 Plotter 실행에 교사의 도움을 받아 수행함. 그래프가 움직이는 것을 관찰하고, 센서를 가리면 그래프가 변한다는 것은 인식함.

### 3차시: 자동 기록 시스템 (CSV 파일 저장)

**S** CSV에 타임스탬프를 추가하는 코드를 자발적으로 작성하고, 30개 데이터 포인트의 평균·최댓값·최솟값을 계산하여 "이 교실의 5분간 온도 범위는 23.1~24.8도, 평균 23.7도"라고 기초 통계 요약을 수행함. 수동 기록과 자동 기록의 차이를 과학 시간의 실험 보고서와 연결하며 "과학 실험에서도 이렇게 자동으로 기록하면 신뢰도가 훨씬 높아질 텐데"라고 교과 간 전이를 보임. CSV 저장에 어려움을 겪는 팀원에게 "파일은 열고, 쓰고, 닫는 세 단계야"라며 핵심 구조를 단순화하여 설명해 주고, 그 팀원이 코드를 완성하도록 옆에서 격려함.

**A** CSV 파일에 타임스탬프를 추가하여 "몇 초째에 온도가 올랐는지"를 시계열로 추적할 수 있도록 코드를 자발적으로 개선함. 30개 데이터 포인트의 평균과 최댓값·최솟값을 계산하며 기초 통계 개념을 활용함. 수동 기록과 자동 기록의 차이를 "표본 간격의 일정성"과 "인간 오차 제거"라는 관점에서 정리하고, 과학 실험에서 자동 기록이 신뢰도를 높이는 이유를 설명함.

**B** DHT20 센서 데이터를 CSV 파일로 자동 저장하는 코드를 작성하며, `open()`, `write()`, `flush()`의 역할을 구분하여 이해함. 5분간 30개 이상의 데이터 포인트를 수집하면서 손으로 센서를 감싸 온도를 올리는 실험을 수행하고, 저장된 CSV에서 해당 시점의 온도 상승을 정확히 확인함. "사람이 직접 기록하면 빠뜨리거나 늦을 수 있는데, 코드는 정확한 간격으로 빠짐없이 저장한다"며 자동 기록의 장점을 과학 실험의 신뢰성과 연결하여 설명함. `for i in range(30)` 반복문의 구조를 이해하고, 측정 횟수를 50으로 변경하여 더 긴 실험을 자발적으로 시도함.

**C** 예제 코드를 참고하여 CSV 저장 코드를 완성하고, 센서 데이터가 파일에 한 줄씩 쌓이는 과정을 확인함. 손으로 센서를 감싸는 실험을 수행하고, CSV를 열어 온도가 올라간 시점을 찾아냄. `for` 반복문이 정해진 횟수만큼 실행된다는 것을 이해하고, `flush()`가 없으면 데이터가 바로 저장되지 않을 수 있다는 것을 교사의 설명으로 납득함. CSV 형식이 쉼표로 값을 구분한다는 구조를 파악함.

**D** 교사의 안내에 따라 코드를 입력하고 CSV 파일이 생성되는 것을 확인함. 파일을 열어 숫자가 저장되어 있다는 것을 확인하나, 데이터에서 특정 시점의 변화를 찾아내는 것은 힌트가 필요함.

**E** 코드 입력에 교사의 도움을 받아 수행함. 코드 실행 후 파일이 만들어졌다는 것을 교사의 설명을 통해 이해하며, 컴퓨터가 데이터를 자동으로 저장할 수 있다는 기본 개념은 인식함.

### 4차시: 첫 번째 대시보드 (DHT20 + 빛 센서 + OLED)

**S** 128x64 OLED 레이아웃을 설계할 때 미술 시간에 배운 시각적 위계(크기, 위치, 대비)를 적용하여 "가장 중요한 온도를 가장 크게, 보조 정보는 작게"라는 디자인 원칙을 설명함. 막대 그래프 길이를 비례식으로 정확히 계산하고, 다른 팀의 대시보드를 보며 "이 팀은 숫자만 나열했는데, 막대 그래프가 있으면 비교가 훨씬 쉬워"라고 시각적 인코딩의 효과를 분석함. 병원 모니터, 자동차 계기판, 날씨 앱의 대시보드 사진을 찾아와 "공통점은 가장 중요한 정보가 가장 눈에 띈다는 것"이라고 정리함. OLED 코드에서 좌표를 잘못 넣어 글자가 겹치는 친구에게 다가가 "x가 가로, y가 세로인데, y를 20씩 띄어봐"라고 좌표계를 설명해 줌.

**A** 128x64 OLED 화면을 영역 분할하여 온도(큰 숫자), 습도(막대 그래프), 조도(아이콘+수치)를 한 화면에 배치하는 레이아웃을 독창적으로 설계함. 막대 그래프 길이를 데이터 값에 비례하도록 계산할 때 비례식을 정확히 적용함. 다른 팀의 디자인과 비교 발표하며 "같은 데이터라도 배치에 따라 의사결정 속도가 달라진다"고 분석하고, 병원 모니터나 자동차 계기판 등 실생활 대시보드와의 공통 설계 원리를 연결하여 이해함.

**B** 3개 I2C 장치(DHT20, 빛 센서, OLED)를 하나의 I2C 버스에 동시 연결하며, 각 장치의 주소(0x38, 0x3C 등)가 다르기 때문에 충돌 없이 작동한다는 원리를 이해함. OLED 128x64 화면에 온도는 큰 숫자로, 습도는 막대 그래프로, 조도는 수치로 표시하는 레이아웃을 직접 설계함. `fill_rect()` 함수로 막대 그래프를 그릴 때 좌표계를 이해하고, 데이터 값에 비례하는 막대 길이를 계산하여 구현함. 다른 팀의 대시보드 디자인과 비교하며 "같은 데이터라도 보여주는 방식에 따라 읽기 쉬움이 달라진다"고 표현하고, 레이아웃을 두 번 이상 수정하며 가독성을 높이려는 반복 개선 과정을 보임.

**C** 3개 장치를 연결하고 OLED에 센서 값을 표시하는 코드를 완성함. 숫자 크기와 위치를 조정하여 온도·습도·조도를 구분해 표시하고, 센서를 조작하면 화면이 실시간으로 바뀌는 것을 확인함. I2C 버스에 여러 장치가 동시에 연결될 수 있다는 것을 이해하며, 1~3차시에서 배운 내용이 하나의 시스템으로 통합되는 경험에 성취감을 표현함.

**D** 교사의 안내에 따라 장치를 연결하고 예제 코드를 실행하여 OLED에 센서 값이 표시되는 것을 확인함. 센서를 조작하면 화면 숫자가 바뀌는 것을 관찰하고 재미를 느낌.

**E** 장치 연결에 교사의 도움을 받아 수행함. 코드 실행 후 OLED 화면에 숫자가 나오고, 센서를 만지면 숫자가 바뀐다는 것을 체험함.

## 2막: 바이브 코딩으로 확장

---

### 5차시: AI에게 코드 시키기 (바이브 코딩 입문)

**S** AI에게 보낸 프롬프트를 세 번에 걸쳐 구체화하며, 요구사항의 정밀도와 코드 품질 사이의 상관관계를 정리하여 "프롬프트 작성 가이드"를 만들어 반 전체에 공유함. 1차시 직접 코딩과 AI 코드를 줄 단위로 비교하여 구조적 차이(함수 분리, 예외 처리, 변수 명명)를 분석하고, 국어 시간의 글쓰기와 연결하며 "AI에게 프롬프트 쓰는 것도 결국 정확하게 글로 표현하는 능력"이라는 교과 간 통찰을 보임. 프롬프트 작성에 어려움을 겪는 친구에게 "먼저 뭘 하고 싶은지 한 문장으로 말해 봐, 그다음에 조건을 하나씩 붙이면 돼"라고 전략을 안내하여 그 친구가 스스로 코드를 생성하는 성공 경험을 갖게 도움.

**A** AI에게 보낸 프롬프트를 세 번에 걸쳐 구체화하며, 요구사항의 정밀도와 코드 품질 사이의 상관관계를 스프레드시트로 정리함. AI가 생성한 코드와 1차시에 직접 작성한 코드를 줄 단위로 비교하여 구조적 차이(함수 분리, 예외 처리 유무, 변수 명명 규칙)를 분석함. "AI는 패턴을 빠르게 조합하지만 하드웨어 맥락을 모르기 때문에, 결국 검증은 사람이 해야 한다"는 결론을 도출하고, 바이브 코딩의 효과적인 사용 조건을 팀원들과 공유함.

**B** AI 챗봇에 "DHT20 센서로 2초마다 온습도를 읽는 MicroPython 코드 만들어줘"라는 프롬프트를 작성하고, 생성된 코드를 1차시에 직접 작성한 코드와 비교함. AI 코드에서 변수명이 더 체계적이고 주석이 포함되어 있음을 발견하는 한편, 센서 주소가 잘못되어 있어 직접 수정하는 경험을 함. 두 번째 프롬프트에서 "쾌적도 판단 + 측정 횟수 카운트 + 10회 후 종료" 조건을 추가하며 요구사항을 구체화할수록 AI 코드의 품질이 높아진다는 것을 깨달음. "AI가 코드를 대신 써주지만, 코드를 읽고 오류를 찾는 건 결국 내가 해야 한다"고 정리하며, 1차시 때 직접 코딩한 경험이 있었기 때문에 AI 코드의 오류를 발견할 수 있었다는 연결고리를 인식함.

**C** AI에게 기본 프롬프트를 작성하여 코드를 받고 Pico에서 실행하여 동작을 확인함. AI가 만든 코드에서 핀 번호 오류를 교사의 힌트와 함께 발견하고 수정함. 두 번째 프롬프트에서 조건을 추가하는 시도를 하고, 프롬프트가 자세할수록 결과가 낫다는 것을 체감함. 바이브 코딩이 코드 작성 시간을 크게 줄여준다는 점을 인식하면서도, 코드를 전혀 모르면 오류를 찾을 수 없다는 것을 이해함.

**D** 교사가 제시한 프롬프트 예시를 참고하여 AI에게 코드를 요청하고 실행함. AI가 만든 코드가 동작하는 것을 확인하고 신기해하나, 코드를 수정하는 것에는 아직 어려움을 느낌.

**E** 교사의 안내로 AI 챗봇에 프롬프트를 복사하여 코드를 받고 실행하는 과정을 체험함. AI가 코드를 만들어 줄 수 있다는 것을 인식함.

### 6차시: 환경 모니터링 대시보드 (CO2 + 온습도 + OLED)

**S** 환기 실험에서 CO2 감소 곡선의 기울기를 분석하여 "처음 2분의 감소율이 가장 크고 이후 완만해진다"는 비선형 패턴을 발견하고, 이를 수학 시간의 감소 함수와 연결하여 설명함. "교실 인원수와 CO2 증가 속도의 관계를 정량적으로 알아보려면, 인원을 바꿔가며 실험해야 한다"고 변인 통제 설계를 제안하며 과학적 사고를 보임. 다른 반 담임교사에게 "우리 반 CO2 데이터를 보여드릴게요, 30분마다 환기하시면 좋을 것 같아요"라며 데이터 기반 제안을 하는 등 교실 밖으로까지 학습이 확장됨. I2C 스캔 코드의 존재를 AI에게 질문해서 알아내고, 센서 주소 충돌 문제를 겪는 두 팀의 디버깅을 도와줌.

**A** 환기 실험 데이터를 시간-CO2 그래프로 스케치하며 감소 곡선의 기울기를 분석하고, "환기 시작 후 처음 2분의 감소율이 가장 크고 이후 완만해진다"는 비선형 패턴을 발견함. 교실 인원수와 CO2 증가 속도의 관계를 가설로 세우고, 다른 학급(인원수 다른)과의 비교 실험을 제안함. AI가 생성한 코드에서 I2C 주소 충돌 가능성을 사전에 예측하고 스캔 코드로 검증하는 체계적 디버깅 절차를 보임.

**B** CO2 센서(SCD41), DHT20, OLED 3개 장치를 I2C 버스에 동시 연결하는 복합 시스템을 바이브 코딩으로 구현함. 창문을 닫고 5분간 CO2 농도 변화를 관찰한 뒤, 창문을 열어 환기했을 때 925ppm에서 600ppm까지 떨어지는 과정을 대시보드로 실시간 확인함. "사람이 25명 있는 교실에서 30분만 지나도 CO2가 1000ppm을 넘는다"는 데이터를 근거로 구체적인 환기 주기(20분마다 5분 환기)를 제안함. AI가 생성한 코드에서 센서 주소 충돌 오류를 발견하고, I2C 스캔 코드를 활용해 실제 연결된 장치 주소를 확인한 뒤 직접 수정하는 등 체계적인 디버깅 과정을 보임.

**C** 바이브 코딩으로 3개 센서 대시보드 코드를 받아 실행하고, 환기 전후 CO2 수치 변화를 관찰함. 창문을 열면 CO2가 내려가고, 닫으면 다시 올라간다는 인과관계를 데이터로 확인하며, "교실 환기가 실제로 공기질에 큰 영향을 미친다"는 결론을 냄. AI 코드에서 발생한 오류를 교사의 힌트와 함께 수정하며, 복합 시스템에서의 디버깅 경험을 쌓음. 4차시 대시보드보다 센서가 하나 더 늘어난 것을 비교하며 시스템 확장의 논리를 이해함.

**D** 교사의 안내에 따라 환기 실험에 참여하고, OLED에 표시되는 CO2 수치가 변하는 것을 관찰함. 환기하면 공기가 좋아진다는 것을 수치로 확인하는 경험을 함.

**E** 교사의 지원으로 장치를 연결하고 코드를 실행함. 환기 실험에 참여하며 화면의 숫자가 바뀌는 것을 관찰하고, 창문을 열면 CO2 수치가 줄어든다는 것을 체험함.

### 7차시: 내 몸의 데이터 (심박 센서 + 파형 대시보드)

**S** 안정 시-운동 직후-1분-2분 회복의 BPM 데이터를 수집하고, 팀원 4명의 회복 곡선을 하나의 표에 정리하여 비교함. "운동을 꾸준히 하는 친구의 회복 속도가 가장 빠르다"는 패턴을 발견하고, 체육 시간에 배운 심폐지구력 개념과 연결하여 "심박 회복 속도가 체력의 객관적 지표가 될 수 있다"고 설명함. THRESHOLD 캘리브레이션의 원리를 "의료 기기도 환자마다 보정하는 것과 같다"고 일반화하며, 보건 시간에 "맥박 센서로 측정하면 손목으로 세는 것보다 정확할 텐데"라고 연결함. 신호가 잘 안 잡히는 친구에게 다가가 "손가락을 살짝만 대고 움직이지 마, 그리고 THRESHOLD를 좀 낮춰봐"라고 하드웨어와 소프트웨어 양쪽의 해결책을 동시에 안내해 줌.

**A** 안정 시-운동 직후-1분 회복-2분 회복의 BPM 데이터를 수집하고, 회복 곡선의 기울기를 비교하여 "회복이 빠른 사람일수록 심폐 기능이 좋다"는 가설을 세움. 팀원 4명의 데이터를 표로 정리하고 운동 습관과의 상관관계를 분석함. THRESHOLD 캘리브레이션이 왜 사람마다 달라야 하는지를 혈류량과 피부 두께 차이로 설명하며, 의료기기의 개인 보정 과정과 연결지어 이해함.

**B** 맥박 센서(PPG)를 연결하고 바이브 코딩으로 심박수 측정 코드를 요청하여, OLED에 실시간 파형 그래프와 BPM을 표시하는 대시보드를 구현함. 안정 시 심박수(72 BPM)를 측정한 뒤 30초간 제자리 뛰기를 하고 직후(112 BPM), 1분 후(95 BPM), 2분 후(78 BPM)의 회복 과정을 체계적으로 기록함. 피크 감지 알고리즘의 THRESHOLD 값을 자신의 손가락 혈류 신호에 맞게 여러 차례 조정하며, "사람마다 최적 값이 다르다"는 캘리브레이션 개념을 체득함. 팀원들의 회복 속도를 비교하여 운동 습관과 심박 회복 속도의 관계에 대해 가설을 세우고, 이를 확인하려면 어떤 추가 데이터가 필요한지 논의함.

**C** 심박 센서를 연결하고 바이브 코딩으로 받은 코드를 실행하여 OLED에 BPM과 파형이 표시되는 것을 확인함. 운동 전후 심박수 차이를 측정하고 기록하며, "쉴 때보다 운동 직후가 40 정도 높고, 2분이면 거의 돌아온다"는 결과를 정리함. THRESHOLD 값이 측정 정확도에 영향을 미친다는 것을 이해하고, 교사의 안내에 따라 값을 조정하여 자신의 신호에 맞춤. 심박 데이터가 건강 모니터링에 활용된다는 실생활 연결을 인식함.

**D** 센서를 손가락에 대고 심박수가 측정되는 것을 확인함. 운동 후 심박수가 올라간다는 것을 직접 경험하고 흥미를 보임. THRESHOLD 조정은 교사가 제시한 값을 사용함.

**E** 교사의 도움으로 센서를 올바르게 접촉하고 코드를 실행함. OLED에 숫자가 나오는 것을 확인하고, 그것이 자신의 심장 박동이라는 것을 이해함.

### 8차시: 스마트 알림 시스템 (초음파 + 부저 + LED Bar)

**S** 거리별 경고 5단계의 임계값을 직접 설계하면서, PWM 주파수-인간 청각 반응 관계를 실험으로 확인하고 "500Hz 이하는 경고 느낌이 약하고, 2000Hz 이상은 불쾌하다"는 최적 범위를 찾아냄. 주차 보조 시스템을 만든 뒤 "시각장애인 보행 보조 장치"와 "교실 복도 혼잡 감지 시스템"이라는 두 가지 사회적 응용 시나리오를 구체적으로 설계하여 기술 시간 담당 교사에게 보여주며 "이런 걸 기술 시간에 만들면 어떨까요"라고 제안함. 4개 장치 연결에 혼란을 겪는 팀에게 "먼저 하나씩 연결해서 각각 테스트하고, 그다음에 합쳐"라는 분할 정복 전략을 알려주고, 그 팀이 완성할 때까지 곁에서 질문에 답해 줌.

**A** 거리별 경고 단계(5단계)의 임계값을 직접 설계하고, 각 단계에서 부저 주파수와 LED 점등 수를 비례적으로 대응시키는 매핑 테이블을 작성함. PWM 주파수와 인간의 청각 반응 관계(높은 주파수 = 긴급감)를 실험으로 확인하고 최적의 경고음을 선정함. 주차 보조 외에 시각장애인용 보행 보조, 교실 출입구 혼잡 감지 등 사회적 문제 해결 시나리오를 구체적으로 설계하여 발표함.

**B** 초음파 거리 센서, 부저, LED Bar, OLED 4개 장치를 조합한 주차 보조 시스템을 바이브 코딩으로 설계하고 구현함. 5cm, 10cm, 20cm, 50cm, 100cm 거리에서 부저 주파수와 LED 점등 개수가 단계적으로 변하는 것을 확인하고, 거리별 경고 기준을 자신이 원하는 값으로 직접 조정함. "하나의 센서 입력이 소리, 빛, 화면 세 가지 출력으로 동시에 나간다"는 구조를 이해하고, PWM으로 부저 주파수를 제어하는 원리를 설명함. 실제 주차장 센서와의 유사점을 찾고, 교실 복도 충돌 방지나 책상 앞 자세 감지 등 자신만의 응용 시나리오를 구체적으로 제안함.

**C** 바이브 코딩으로 받은 코드를 실행하여 물체가 가까워지면 부저가 빨라지고 LED가 늘어나는 것을 확인함. 여러 거리에서 테스트하며 경고 단계를 체험하고, 센서 하나로 여러 출력을 제어할 수 있다는 것을 이해함. 실제 주차장에서 비슷한 원리의 시스템이 쓰인다는 점을 인식하고, 코드에서 거리 임계값을 한두 군데 바꿔보는 시도를 함.

**D** 교사의 안내에 따라 장치를 연결하고 코드를 실행함. 손을 가까이 가져가면 소리가 빨라지는 것을 체험하고, 실제 주차장 센서와의 유사성을 인식함.

**E** 교사의 지원으로 장치를 연결하고 코드를 실행함. 손을 가까이 하면 소리가 나고, 멀리 하면 소리가 멈춘다는 기본 동작을 체험함.

### 9차시: 데이터 정리와 시각화 (PC Python + pandas + matplotlib)

### 10차시: AI 데이터 분석 (AI에게 데이터 해석 요청)

AI가 교실 크기를 모르고 한 잘못된 추론을 짚어낸 뒤, 보완 정보를 추가한 프롬프트에서 분석이 개선되는 것을 검증하고, "이건 국어 시간에 배운 논증과 같아요 — 근거가 부족하면 주장이 약해진다"고 교과 간 연결을 보임. "AI 분석의 신뢰성을 판단하려면 해당 분야의 기본 지식이 필수"라는 원칙을 정리하여 반 전체에 공유하고, 프롬프트 작성에 어려움을 겪는 친구들에게 "맥락 정보를 빠뜨리지 마, AI는 우리 교실을 본 적이 없으니까"라며 핵심 전략을 알려줌.

**A** 동일 데이터에 대해 3가지 서로 다른 프롬프트(패턴 탐색, 인과 분석, 예측)를 작성하고, 각각의 AI 응답을 비교하여 "질문의 프레이밍이 분석 결과를 결정한다"는 통찰을 도출함. AI가 맥락 부족(교실 크기, 계절, 냉난방 여부)으로 잘못된 추론을 한 사례를 정확히 짚어내고, 보완 정보를 추가한 후속 프롬프트에서 분석 품질이 향상되는 것을 검증함. "AI 분석의 신뢰성을 판단하려면 해당 분야의 기본 지식이 필수"라는 원칙을 정리하여 팀원들과 공유함.

**B** 수집한 센서 데이터를 텍스트로 정리하여 AI에게 분석을 요청하되, "25명 교실, 30초 간격 측정, 5분 시점에 환기 시작"이라는 맥락 정보를 함께 제공함. AI가 발견한 패턴(환기 후 CO2 감소 속도, 온습도 상관관계 등)을 검토하며 "정확한 분석"과 "과도한 추측"을 구분함. AI가 교실 크기, 창문 방향, 계절 등을 알 수 없어 잘못된 가정을 하는 경우를 발견하고, "AI는 빠르게 패턴을 찾지만 현장의 맥락은 사람만이 제공할 수 있다"고 정리함. 동일 데이터에 대해 프롬프트를 달리하면 분석 결과가 달라지는 것을 확인하며, 질문 설계가 AI 활용의 핵심임을 인식함.

**C** AI에게 맥락 정보와 함께 데이터를 보내고 분석 결과를 받아 읽음. AI가 찾은 패턴(환기 후 CO2 감소 등)을 이해하고, 일부 분석이 실제 교실 상황과 맞는지 대조하여 확인함. AI의 장점(빠른 분석, 패턴 요약)과 한계(맥락을 모름, 가끔 추측이 과함)를 구분할 수 있으며, "AI가 말하는 것을 다 믿으면 안 되고, 우리가 알고 있는 사실과 비교해야 한다"는 인식을 가짐.

**D** 교사의 프롬프트 예시를 참고하여 AI에게 데이터 분석을 요청하고, 결과를 읽어봄. AI가 요약한 내용이 환기 실험과 관련 있다는 것은 이해하나, 분석의 정확성을 독립적으로 판단하기는 어려움.

**E** 교사의 지원으로 AI에게 데이터를 보내고 응답을 받는 과정을 체험함. AI가 데이터를 보고 무언가를 분석해 준다는 것을 인식함.

### 11차시: 팀 프로젝트 설계

**S** 브레인스토밍에서 10개 이상의 문제를 제안하고, 각 문제에 대해 "센서 가용성, 데이터 수집 난이도, 사회적 영향력"을 기준으로 평가 매트릭스를 만들어 팀의 의사결정을 체계적으로 이끔. **A** 브레인스토밍에서 10개 이상의 문제를 제안하고, 각 문제에 대해 "센서 가용성, 데이터 수집 난이도, 사회적 영향력"을 기준으로 평가 매트릭스를 만들어 팀의 의사결정을 이끔. 프로젝트 명세에 데이터 수집 계획(샘플 수, 주기, 기간)과 예상 결과를 구체적으로 포함시키며, 바이브 코딩과 직접 코딩의 경계를 "센서 드라이버는 AI, 분석 로직은 직접"으로 명확히 구분하여 전략을 수립함.

**E** 팀 활동에 참여하며 분위기를 파악함. 자신의 역할이 무엇인지 팀원이나 교사의 설명을 통해 이해하고, 다음 차시에서 수행할 최소한의 과제를 인지함.

### 12차시: 실시간 대시보드 제작

**S** OLED와 PC 대시보드를 시리얼 통신으로 연동하면서, 팀원들에게 "데이터 형식은 `센서1,센서2,센서3\n`으로 통일하자. 그래야 PC 코드에서 split으로 파싱할 수 있어"라며 인터페이스 표준을 제안하여 통합 오류를 사전에 방지함. `deque` 버퍼 크기를 50, 100, 200으로 바꿔가며 "표시 기간과 메모리 사용의 균형"을 실험하고, 정보 시간에 배운 "모듈화"와 연결하여 "각자 맡은 코드가 모듈이고, 인터페이스만 맞추면 합칠 수 있다"고 설명함. 대시보드 완성이 늦어지는 팀의 시리얼 통신 오류를 진단해 주고("baud rate가 안 맞아"), 자기 팀 작업과 병행하며 두 팀 모두 완성하도록 도움.

**A** OLED 대시보드와 PC 대시보드를 시리얼 통신으로 연동하되, 데이터 전송 실패 시 재전송하는 예외 처리를 자발적으로 추가함. 실시간 그래프의 `deque` 버퍼 크기를 조절하며 "표시 기간과 메모리 사용의 균형"을 실험하고, 팀원들에게 각자 맡은 코드를 통합할 때의 인터페이스(데이터 형식, 전송 주기)를 미리 합의하자고 제안하여 통합 오류를 사전에 방지함.

**B** 팀 프로젝트에 맞는 복합 센서 대시보드를 바이브 코딩으로 구현하되, AI 코드의 센서 주소, 포트 번호, 화면 레이아웃 오류를 직접 수정함. OLED 대시보드와 PC Python 대시보드를 시리얼 통신으로 연동하여, Pico에서 측정한 데이터가 실시간으로 PC 그래프에 반영되는 이중 대시보드를 완성함. `deque` 버퍼를 활용한 실시간 그래프 갱신 구조를 이해하고, 센서 읽기 실패 시 예외 처리를 추가하여 시스템 안정성을 높임. 여러 시스템을 통합할 때 발생하는 타이밍 이슈와 데이터 동기화 문제를 경험하고, 테스트와 반복 개선의 중요성을 체감함.

**C** 바이브 코딩으로 받은 대시보드 코드를 팀 센서에 맞게 수정하여 OLED에 데이터를 표시하고, PC 대시보드와의 연동은 교사의 안내를 받아 완성함. 코드 오류 발생 시 에러 메시지를 읽고 원인을 추측할 수 있으며, 팀 내에서 자신이 맡은 파트를 책임감 있게 완수함. 두 대시보드가 동시에 동작하는 것을 확인하고 성취감을 표현함.

**D** 팀에서 맡은 부분을 교사의 도움과 함께 완성함. 전체 시스템이 동작하는 것을 확인하고 성취감을 느낌.

**E** 팀원들의 작업을 관찰하며 도움이 필요한 부분(부품 정리, 테스트 보조)에서 기여함. 최종 대시보드가 동작하는 것을 팀과 함께 확인함.

### 13차시: 센서 데이터로 ML 입문

**S** Teachable Machine에서 소리 분류 모델을 학습시킨 뒤, 박수와 노크의 혼동 원인을 "두 소리 모두 짧은 충격음이라 주파수 스펙트럼이 유사하다"고 분석하고, 녹음 샘플의 다양성(강한 박수/약한 박수, 문 노크/책상 노크)을 높였을 때 정확도가 올라가는 것을 실험으로 검증함. "학습 데이터에 편향이 있으면 모델도 편향된다"는 원리를 과학 시간의 "표본의 대표성" 개념과 연결하여 설명함. ML 체험에 어려움을 겪는 친구에게 "넌 지금 AI 선생님이야. 학생에게 예시를 많이 보여줄수록 잘 배우는 것처럼, 데이터를 많이 줄수록 모델이 잘 배워"라는 비유로 핵심 원리를 전달하여 그 친구도 모델을 성공적으로 학습시키게 도움.

**A** Teachable Machine에서 소리 분류 모델을 학습시킨 뒤, 박수와 노크의 혼동 원인을 "두 소리 모두 짧은 충격음이라 주파수 스펙트럼이 유사하다"고 분석하고, 녹음 샘플을 더 다양하게 수집하면 정확도가 올라가는지 실험하여 검증함. "규칙 기반 분류는 조건문이 기하급수적으로 늘어나지만 ML은 데이터에서 경계를 자동으로 찾는다"는 차이를 구체적 예시와 함께 설명하고, 학습 데이터의 편향이 결과에 미치는 영향까지 논의함.

**B** Google Teachable Machine에서 박수, 노크, 말소리 3가지 소리 클래스를 각각 8초 이상 녹음하고 모델을 학습시킴. 새로운 소리로 테스트하며 분류 정확도를 확인하고, 박수와 노크가 혼동되는 경우의 원인(비슷한 충격음 패턴)을 분석함. "규칙 기반으로 소리를 분류하려면 조건문이 끝없이 늘어나지만, ML은 데이터에서 패턴을 스스로 학습한다"는 차이를 이해하고, 이것이 왜 중요한지를 설명함. 가속도 센서 데이터를 정지/걷기/흔들기로 라벨링하여 CSV로 저장하며, "좋은 ML 모델에는 충분한 양의 정확히 라벨링된 데이터가 필요하다"는 데이터 품질의 중요성을 인식함. 신뢰도 점수(confidence score)를 해석하며 모델이 "확신하는 분류"와 "애매한 분류"를 구분하는 비판적 사고를 보임.

**C** Teachable Machine에서 소리 분류 모델을 만들고 테스트하여 대부분 정확하게 분류되는 것을 확인함. 일부 오분류가 발생하는 이유를 "소리가 비슷해서"로 설명하고, 녹음을 더 많이 하면 정확도가 올라갈 수 있다고 추측함. ML이 사람이 만든 규칙보다 유연하다는 것을 이해하며, 가속도 데이터 라벨링에도 참여하여 "라벨을 잘못 붙이면 모델이 잘못 배운다"는 점을 인식함.

**D** 교사의 안내에 따라 소리를 녹음하고 모델을 학습시킴. 자신의 박수 소리를 모델이 맞추는 것을 확인하고 신기해함. ML이 데이터로부터 배운다는 기본 개념을 이해함.

**E** 교사의 지원으로 소리 녹음과 모델 학습을 체험함. 컴퓨터가 소리를 듣고 구분할 수 있다는 것을 인식함.

### 14차시: 발표와 피드백

**S** 발표를 "문제 → 데이터 → 분석 → 제안"의 논리적 흐름으로 구성하고, 라이브 시연 중 예상치 못한 수치 변동을 즉석에서 "지금 창문이 열려 있어서 CO2가 급격히 떨어지고 있는 거예요"라고 해석하며 청중의 이해를 돕음. 다른 팀에게 "데이터 수집 기간을 3일로 늘리면 일별 패턴이 보일 거예요", "LED 경고를 3단계에서 5단계로 세분화하면 사용자가 더 직관적으로 느낄 거예요"와 같이 구체적이고 실행 가능한 개선안을 제시함. 발표 시 긴장하는 팀원에게 "네가 만든 부분을 네가 가장 잘 알아. 그냥 뭘 했는지 편하게 말해"라고 격려하여 팀 전체의 발표 품질을 높이고, 국어 시간에 배운 "주장-근거-정리"의 논증 구조가 발표에도 적용된다고 연결함.

**A** 발표에서 문제 정의 → 데이터 시연 → 분석 → 시사점의 논리적 흐름을 명확히 구성하고, 라이브 대시보드 시연 중 예상치 못한 수치 변동을 즉석에서 해석하여 설명함. 다른 팀의 발표에 대해 5개 항목별 점수뿐 아니라 "데이터 수집 기간을 늘리면 신뢰도가 높아질 것", "LED 경고 단계를 3단계에서 5단계로 세분화하면 사용성이 좋아질 것"과 같은 구체적이고 실행 가능한 개선안을 제시함. 피드백 집계 코드를 작성하여 전체 팀의 강점 분포를 시각화하고, "기술 구현과 전달력은 별개의 역량"이라는 점을 데이터로 보여줌.

**B** 5분 발표를 구조적으로 구성함: 문제 발견(30초) → 하드웨어 시연(1.5분) → 데이터 인사이트(1분) → ML 결과(1분) → 시사점(1분). OLED 대시보드와 PC 대시보드를 라이브로 시연하며 실시간 데이터 변화를 청중에게 보여줌. "CO2가 45분 시점에 1000ppm을 초과했다"와 같이 구체적 데이터를 근거로 주장을 뒷받침함. 다른 팀의 발표에 대해 대시보드 디자인, 데이터 신뢰성, ML 활용도, 발표 명료성, 실용성 5개 항목을 점수로 평가하고, "센서 위치를 바꾸면 더 정확한 데이터를 얻을 수 있을 것"과 같은 건설적인 개선 의견을 작성함. "기술적 구현만큼 소통과 설득도 중요하다"는 점을 체감함.

**C** 발표에서 팀 프로젝트의 주요 내용을 전달하고 대시보드를 시연함. 발표 시간 배분을 대체로 지키며, 데이터를 활용한 설명을 시도함. 다른 팀의 발표에 대해 항목별 점수를 차등적으로 매기고, "대시보드 디자인이 깔끔했다", "센서 데이터가 안정적이었다" 등 구체적 근거를 가진 짧은 피드백을 작성함.

**D** 팀 발표에서 자신이 맡은 부분(예: 하드웨어 시연)을 수행함. 다른 팀의 발표를 듣고 점수를 매기나, 구체적 피드백 작성은 짧음. 발표 경험 자체에서 의미를 찾음.

**E** 팀 발표에 참석하여 함께 서고, 다른 팀의 발표를 관람함. 점수 기입에 참여하며, 발표와 피드백이라는 소통 활동을 체험함.

### 15차시: 되돌아보기와 다음 단계

**S** 15주간의 성장을 "1차시에는 코드 한 줄이 무서웠는데, 지금은 AI에게 시스템을 설계시키고 내가 검증한다"라는 구체적 에피소드와 함께 서술하고, 센서 패스포트에 각 센서별 성공과 실패 경험을 모두 기록함. 14가지 Python 개념 중 부족한 영역을 솔직히 표시하며 코스 B에서의 구체적 학습 계획을 세움. "센서 → 데이터 → 시각화 → AI → ML"이라는 파이프라인이 환경 모니터링, 건강 관리, 스마트 농업, 재난 감지 등에 적용 가능하다는 전이를 구체적 예시와 함께 설명하고, "수학 시간의 통계, 과학 시간의 실험, 국어 시간의 논증, 기술 시간의 제작이 이 수업에서 다 연결됐다"고 교과 통합적 성찰을 표현함. 뒤돌아보기 작성이 막막한 친구에게 "1막에서 뭐 만들었는지부터 하나씩 떠올려 봐. 센서 처음 꽂았을 때 기억나지?"라고 회상을 도와주며 모든 학생이 성찰을 완성하도록 이끔.

**A** 15주간의 성장을 "1차시에는 코드 한 줄을 이해하는 데 10분이 걸렸는데, 이제는 AI에게 시스템을 설계시키고 결과를 검증한다"와 같이 구체적 에피소드와 함께 서술함. 센서 패스포트에 각 센서별 핵심 발견과 실패 경험을 모두 기록하고, 14가지 Python 개념의 자기 평가에서 부족한 영역을 솔직히 표시하며 코스 B에서의 학습 계획을 구체적으로 세움. 전체 과정을 "센서 → 데이터 → 시각화 → AI → ML"이라는 파이프라인으로 요약하고, 이 구조가 환경 모니터링, 건강 관리, 스마트 농업 등 일상의 다양한 문제에 적용 가능하다는 전이 가능성을 구체적 예시와 함께 설명함.

**B** 15주간의 학습 여정을 5개 성찰 질문에 대해 구체적으로 기술함. 센서 패스포트를 완성하여 사용한 모든 센서, 측정 횟수, 핵심 발견을 체계적으로 정리함. 14가지 Python 핵심 개념(변수, 반복문, 조건문, 함수, 파일 I/O, 딕셔너리 등)의 습득 여부를 자가 점검하고, 1차시와 현재의 역량을 비교하는 성장 시각화 코드를 작성함. "처음에는 코드 한 줄이 무서웠는데, 이제는 AI에게 복잡한 시스템을 요청하고 결과를 검증할 수 있게 되었다"고 자신의 성장을 메타인지적으로 표현하며, 코스 B에서 도전하고 싶은 구체적 프로젝트 아이디어를 제시함.

**C** 성찰 질문에 답하며 인상 깊었던 차시와 그 이유를 구체적으로 기술함. 센서 패스포트를 대부분 완성하고, 사용한 센서별 핵심 활동을 기록함. Python 개념 중 익숙한 것(변수, 반복문, 조건문)과 아직 어려운 것을 솔직히 구분하며, 15주 전의 자신과 비교하여 "이제 코드를 보면 대략 무슨 뜻인지 알 수 있고, AI에게 원하는 것을 설명할 수 있다"는 변화를 인식함. 코스 B에 관심을 보이며 어떤 레벨부터 시작할지 고민함.

**D** 성찰 질문에 짧게 답하며, 가장 재미있었던 활동과 어려웠던 활동을 기록함. 센서 패스포트를 부분적으로 작성하고, 자신이 처음보다는 성장했다는 것을 느낌.

**E** 교사의 질문에 따라 기억나는 활동을 하나 말하고, 15주를 마친 것에 대해 성취감을 표현함. 센서로 데이터를 읽고 AI를 활용할 수 있다는 경험 자체가 의미 있는 학습이 됨.

### 1막 종합 (1~4차시): 센서와 친해지기

**S** 센서 연결부터 대시보드까지의 전체 파이프라인을 직접 코딩으로 구현하며, 각 단계의 입출력 관계를 설명할 뿐 아니라 다른 교과와의 연결을 자발적으로 발견함. 수학 시간의 비례·통계, 과학 시간의 변인 통제와 측정 오차, 미술 시간의 시각적 위계를 이 수업의 맥락에 자연스럽게 적용하며, "결국 센서가 하는 건 과학 실험의 측정과 같다"고 교과 간 공통 구조를 파악함. 코드나 연결에 어려움을 겪는 친구들에게 핵심 개념을 단순화하여 설명해 주고, 모든 팀이 활동을 완료하도록 교실 안에서 자연스럽게 돕는 모습이 일관됨.

**A** 센서 연결, 데이터 수집, 파일 저장, 대시보드 시각화의 전체 데이터 파이프라인을 직접 코딩으로 구현하며, 각 단계의 입출력 관계를 정확히 설명할 수 있음. 측정값의 분포와 변이를 관찰하며 통계적 사고의 싹을 보이고, "동일 환경에서도 측정 위치에 따라 값이 달라진다"는 변인 통제의 필요성을 인식함. 대시보드 레이아웃 비교를 통해 정보 디자인 감각을 보여주고, 측정 간격 변경이나 센서 조합 추가 등 교재 범위를 넘는 실험을 자발적으로 설계함. 팀원들에게 측정 위치 분담을 제안하거나 결과를 비교 발표하는 등 협업을 주도하는 모습이 돋보임.

**B** 센서 연결부터 대시보드까지의 전체 흐름을 이해하고, 코드의 각 줄이 하드웨어 동작에 직접 대응한다는 것을 체득함. 추상적인 숫자를 그래프와 대시보드로 시각화했을 때 정보의 가독성이 높아지는 경험을 통해 데이터 표현 방식의 중요성을 스스로 인식함. I2C 통신, ADC 변환, 파일 I/O 등 하드웨어-소프트웨어 인터페이스의 기초를 탄탄히 다지며, 코드의 일부를 자발적으로 변경(sleep 값, 반복 횟수 등)하여 결과 차이를 관찰하는 탐구적 자세를 보임. 팀원들과 측정 결과를 공유하고 차이의 원인을 논의하며 데이터 기반 소통 능력을 발전시킴.

**C** 각 차시의 핵심 활동을 완수하고 센서 데이터 수집과 시각화의 기본 흐름을 이해함. 코드를 따라 작성하고 실행하는 능력이 안정적이며, 센서 값이 변하는 이유를 물리적 원인과 연결하여 설명할 수 있음. CSV 파일과 OLED 대시보드의 역할 차이를 이해하고, 그래프 읽기에 자신감을 가짐. 간단한 코드 수정(값 변경)을 시도하며, 결과를 정리하여 발표하는 데 참여함.

**D** 교사의 안내에 따라 각 차시의 활동을 수행하고, 센서가 환경을 측정하고 코드가 이를 처리한다는 기본 개념을 이해함. 직접 코딩에 재미를 느끼기 시작하며, 센서 조작 시 화면이나 파일에 변화가 생기는 것을 관찰하고 흥미를 보임.

**E** 교사의 지원을 받아 센서 연결과 코드 실행을 체험함. 센서가 환경 데이터를 읽고 코드가 이를 화면에 보여줄 수 있다는 기본 경험을 하며, 기술에 대한 친숙함을 형성함.

### 2막 종합 (5~10차시): 바이브 코딩으로 확장

**S** 바이브 코딩을 "인간-AI 협업 프로세스"로 이해하고, 프롬프트 설계 → 코드 검토 → 디버깅 → 요구사항 반복의 체계적 워크플로를 확립하여 다른 친구들에게도 전파함. 복합 센서 시스템에서 CO2-환기, 심박-운동의 인과관계를 데이터로 검증하며 과학적 탐구 과정을 일관되게 실천하고, 이를 과학·수학·국어 등 다른 교과의 개념과 자연스럽게 연결함. 데이터 정제에서 통계적 도구를 목적에 맞게 선택하고, AI 분석 결과에 대해 현장 맥락을 기반으로 비판적 판단을 내림. 어려움을 겪는 친구들에게 비유나 단순화된 설명으로 핵심을 전달하여, 반 전체의 학습 수준을 끌어올리는 데 기여함.

**A** 바이브 코딩을 단순 코드 생성이 아닌 "인간-AI 협업 프로세스"로 이해하고, 프롬프트 설계 → 코드 검토 → 디버깅 → 요구사항 반복의 체계적 워크플로를 확립함. 복합 센서 시스템에서 발생하는 변수 간 상관관계(CO2와 환기, 심박과 운동)를 분석하고, 가설을 세워 추가 실험을 제안하는 과학적 사고가 일관됨. 데이터 정제에서 통계적 도구(이상치 기준, 이동평균)를 목적에 맞게 선택하고 그 근거를 설명하며, AI 분석 결과에 대해 현장 맥락을 기반으로 비판적 판단을 내림. 기술적 역량과 함께 "AI를 잘 쓰려면 그 분야를 먼저 알아야 한다"는 메타 인식이 뚜렷함.

**B** AI에게 자연어로 코드를 요청하고, 생성된 코드를 읽고 오류를 수정하고 요구사항을 변경하는 과정을 반복하며 바이브 코딩의 효과적인 활용법을 체득함. 단일 센서에서 복합 센서 시스템(CO2+온습도, 심박, 초음파+부저+LED)으로 확장하고, 하드웨어 대시보드에서 PC 데이터 분석과 AI 해석까지 영역을 넓히며, "AI는 코드를 생성하는 도구이고, 코드의 품질과 의미를 판단하는 것은 인간의 몫"이라는 핵심 원칙을 내면화함. 데이터 정제 과정에서 이상치 제거와 이동평균의 효과를 직접 비교하고, AI 분석 결과에서 정확한 패턴과 과도한 추측을 구분하는 등, 데이터 리터러시와 AI 리터러시를 동시에 발전시킴. 팀원들의 데이터와 비교하며 가설을 세우고 논의하는 과학적 태도를 보임.

**C** 바이브 코딩으로 코드를 생성하고 기본적인 오류를 교사의 힌트와 함께 수정할 수 있음. 다양한 센서(CO2, 심박, 초음파)와 대시보드를 경험하며 활용 범위가 넓어지고, 프롬프트를 구체적으로 쓸수록 결과가 좋아진다는 것을 체감함. AI의 장점과 한계를 구분하며, 데이터 정제와 시각화의 필요성을 이해함. 환기 실험, 심박 측정 등에서 인과관계를 데이터로 확인하는 경험을 쌓고, 이동평균의 수학적 원리를 이해함.

**D** 바이브 코딩을 통해 직접 코딩보다 복잡한 시스템을 경험하고, AI가 코드 작성을 도와줄 수 있다는 것을 체감함. 환기 실험이나 심박 측정 등 흥미로운 활동에 적극 참여하며, 코드 수정에는 아직 교사의 도움이 필요하나 코드의 대략적 의미는 파악하기 시작함.

**E** AI에게 요청하면 코드가 나오고 이를 실행하면 장치가 동작한다는 전체 흐름을 체험함. 환기, 심박, 거리 측정 등 다양한 실험에 참여하며, 센서와 코드가 실생활 문제와 연결된다는 것을 인식함.

### 3막 종합 (11~15차시): 대시보드와 머신러닝

**S** 문제 발견부터 ML 적용, 발표까지의 전체 공학 설계 과정을 주도하면서, 자기 팀의 완성도를 높이는 동시에 다른 팀의 어려움을 진단하고 해결을 도움. ML 모델의 성능을 데이터 양·다양성·라벨링 정확도와 연결지어 분석하고, "표본의 대표성"이나 "논증의 근거" 같은 다른 교과 개념을 자연스럽게 적용함. 발표에서 데이터 기반 논증으로 청중을 설득하고, 다른 팀에게 실행 가능한 개선안을 제시함. 15주간의 학습을 하나의 파이프라인으로 통합하여 새로운 문제에 독립적으로 적용할 수 있는 전이 역량을 갖추고, 이 구조가 수학·과학·국어·기술·사회 등 여러 교과에 걸쳐 작동한다는 통합적 시야를 보임.

**A** 문제 발견부터 ML 적용, 발표까지의 전체 공학 설계 과정을 주도하며, 설계 단계에서의 의사결정(센서 선택, 데이터 전략, 역할 분담)이 최종 산출물의 품질에 직접 영향을 미친다는 것을 체험적으로 이해함. ML 모델의 성능을 데이터 양·다양성·라벨링 정확도와 연결지어 분석하고, 데이터 편향이 모델 성능에 미치는 영향을 실험으로 검증함. 발표에서 데이터 시각화와 논리적 구조로 청중을 설득하고, 다른 팀에 대해 실행 가능한 개선안을 제시하는 등 비판적이면서도 건설적인 피드백 능력을 보임. 15주간의 학습을 하나의 파이프라인으로 통합하여 새로운 문제에 독립적으로 적용할 수 있는 전이 역량을 갖춤.

**B** 팀 프로젝트의 설계·구현·발표를 경험하며 기술적 역량과 협업 역량을 통합적으로 발휘함. 프로젝트 명세를 구체적으로 정의하고, 바이브 코딩과 직접 코딩의 역할을 구분하여 전략을 세움. 이중 대시보드(OLED+PC)를 완성하고 통합 과정에서의 오류를 디버깅함. ML의 기본 원리(수집→라벨링→학습→추론)를 실제 데이터로 실습하여 이해하고, 모델의 정확도와 데이터 품질의 관계를 이해함. 발표에서 구체적 데이터를 근거로 주장을 뒷받침하고, 다른 팀의 작업에 건설적 피드백을 제공함. 15주간 축적한 역량을 종합하여 독립 프로젝트를 설계할 수 있는 수준에 도달하며, 코스 B에 대한 구체적 관심을 보임.

**C** 팀에서 자신의 역할을 책임감 있게 수행하여 프로젝트 완성에 실질적으로 기여함. ML이 데이터에서 패턴을 학습한다는 개념을 이해하고, Teachable Machine으로 분류 모델을 만들어 테스트함. 발표에 적극 참여하고, 다른 팀의 발표에 구체적 근거가 있는 피드백을 작성함. 15주간의 성장을 인식하고, 센서-코딩-AI의 연결 구조를 설명할 수 있음.

**D** 팀 활동에 참여하여 맡은 부분을 완수하고, 팀 프로젝트가 완성되는 전체 과정을 경험함. ML과 대시보드의 기본 개념을 인식하며, 발표를 통해 다른 팀의 접근을 관찰하고 배움.

**E** 팀과 함께 프로젝트를 완성하는 과정에 참여하고, 15주간의 수업을 마무리한 성취감을 느낌. 센서, 코드, AI, ML이라는 도구들이 존재한다는 것을 인식함.

## 전체 종합 (15차시 전 과정)

---

**S** 15주간의 수업을 통해 "센서 → 데이터 → 코드 → 시각화 → AI → ML"의 전체 파이프라인을 독립적으로 설계하고 구현할 수 있는 역량을 갖추며, 이 구조가 학교 안팎의 다양한 문제에 적용 가능하다는 전이를 실제 사례와 함께 설명함. 직접 코딩으로 기초를 다진 뒤 바이브 코딩으로 복잡한 시스템을 구축하는 전환을 능숙하게 수행하고, AI 코드의 품질을 비판적으로 검토하며 현장 맥락에 맞게 수정하는 능력이 돋보임. 수학의 통계·비례, 과학의 실험 설계·변인 통제, 국어의 논증 구조, 기술의 제작 과정, 사회의 문제 해결 방법론 등 여러 교과에서 배운 개념을 이 수업의 맥락에 자연스럽게 적용하며, "결국 이 모든 교과가 하나의 프로젝트에서 만난다"는 통합적 사고를 보여줌. 수업 내내 어려움을 겪는 친구들에게 핵심을 비유로 설명해 주거나 디버깅을 도와주며, 자기 자신의 학습뿐 아니라 반 전체의 학습 수준을 끌어올리는 데 기여함. 코스 B 이상의 독립 프로젝트를 구체적으로 구상하고, 센서 기반 문제 해결이 자신의 진로 탐색과도 연결될 수 있다는 가능성을 표현함.

**A** 직접 코딩(1막)에서 바이브 코딩(2막), 팀 프로젝트와 ML(3막)로 이어지는 전체 학습 곡선을 충실히 따라가며, 각 단계에서 핵심 역량을 체득함. 센서를 연결하고 데이터를 수집·저장·시각화하는 통합 능력을 갖추고, AI에게 프롬프트를 작성하여 코드를 생성한 뒤 오류를 체계적으로 수정하는 바이브 코딩 워크플로를 능숙하게 수행함. 데이터의 통계적 처리를 목적에 맞게 적용하고, AI 분석 결과에서 현장 맥락을 기반으로 비판적 판단을 내림. 팀 프로젝트에서 설계를 주도하고 역할을 조율하며, 발표에서는 데이터 기반 논증으로 청중을 설득하고 다른 팀에게 구체적 개선안을 제시함. 측정 간격 변경, 추가 실험 설계, 사회적 응용 시나리오 제안 등 교재 범위를 넘는 탐구가 일관되며, "센서 → 데이터 → AI → ML" 구조를 다양한 맥락에 전이할 수 있다는 확신과 함께 코스 B 이상의 프로젝트를 구체적으로 구상함.

**B** 15주간의 수업을 통해 센서 연결부터 대시보드, 바이브 코딩, 데이터 분석, ML까지의 전체 흐름을 안정적으로 수행할 수 있게 됨. 1막에서 코드의 각 줄이 하드웨어 동작에 대응한다는 것을 체득하고, 2막에서 AI를 활용한 코드 생성·오류 수정·데이터 정제를 경험하며, 3막에서 팀 프로젝트의 설계-구현-발표를 완수함. "AI는 코드를 생성하는 도구이고 판단은 인간의 몫"이라는 원칙을 내면화하고, ML이 데이터에서 패턴을 학습한다는 기본 원리를 실제 데이터로 이해함. 이동평균, 이상치 제거 등 통계적 도구의 원리를 수학적으로 파악하고, 발표에서 구체적 데이터를 근거로 주장을 뒷받침하며, 다른 팀에 건설적 피드백을 제공함. "처음에는 코드 한 줄이 무서웠는데, 이제는 AI에게 복잡한 시스템을 요청하고 결과를 검증할 수 있다"는 성장을 메타인지적으로 표현하며, 새로운 프로젝트에 도전할 준비가 됨.

**C** 15주간 센서를 연결하고, 코드를 실행하고, 데이터를 수집하고, AI를 활용하고, ML을 체험하는 전체 과정에 참여하며, 피지컬 컴퓨팅과 AI의 기본 흐름을 형성함. 1막에서 직접 코딩의 재미를 느끼고, 2막에서 바이브 코딩으로 더 복잡한 시스템을 경험하며, 3막에서 팀 프로젝트에 기여함. AI의 장점과 한계를 구분하고, 인과관계를 데이터로 확인하는 경험을 쌓음. 팀 내에서 자신의 역할을 수행하고, 발표와 피드백에 참여하며, "처음보다 코드와 센서가 훨씬 익숙해졌다"는 성장을 구체적으로 인식함.

**D** 15주간의 수업에 참여하며 센서, 코드, AI, ML이라는 도구를 체험함. 교사의 안내에 따라 각 차시의 활동을 수행하고, 특히 흥미를 느낀 활동(환기 실험, 심박 측정, 소리 분류 등)에서 적극적인 반응을 보임. 코드 수정에는 도움이 필요하나, 코드의 대략적 의미를 파악하고 센서 데이터의 변화를 관찰할 수 있게 됨. 팀 활동에 참여하며 협업의 가치를 경험함.

**E** 15주간의 수업에 참여하며 센서, 코드, AI, 머신러닝이라는 기술이 존재하고 이들이 연결될 수 있다는 것을 인식함. 교사의 지원을 받아 활동에 참여하고, 코드를 실행하면 장치가 동작한다는 기본 경험을 함. 수업을 끝까지 완주한 것 자체가 의미 있는 성취이며, 기술에 대한 친숙함이 형성됨.
