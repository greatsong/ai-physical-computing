## 활동 가이드

### Step 1. 문제 발견 (10분)

팀원 3~4명이 모여서, 학교 안에서 **센서+대시보드+AI로 해결하면 좋겠다** 싶은 문제를 자유롭게 적어보세요.

**브레인스토밍 규칙:**
- 비판 금지 -- 어떤 아이디어든 일단 적어요
- 자유롭게 -- 엉뚱한 것도 환영해요
- 연결하기 -- 다른 사람의 아이디어에 덧붙여도 좋아요
- 최소 10개 -- 숫자 목표를 정해두면 더 많이 나와요

**예시 프로젝트 아이디어:**

| 프로젝트 | 센서 | 대시보드 | AI 활용 |
|----------|------|----------|---------|
| 교실 환경 종합 대시보드 | 온습도 + CO2 + 빛 | OLED에 실시간 쾌적도 표시 | 환기 시점 예측 |
| 급식실 소음 모니터 | 소리 센서 | 소음 레벨 막대그래프 | 시간대별 패턴 분류 |
| 복도 통행량 측정 | 초음파 센서 | 시간대별 통행량 차트 | 혼잡 시간 예측 |

### Step 2. 센서/대시보드 설계 (15분)

브레인스토밍에서 가장 흥미로운 아이디어를 하나 골라, 구체적으로 설계해보세요.

**설계 체크리스트:**
- [ ] 어떤 센서를 사용할 것인가? (1~3개)
- [ ] OLED 대시보드에 무엇을 표시할 것인가? (숫자? 그래프? 상태 아이콘?)
- [ ] PC 대시보드도 만들 것인가? (웹 대시보드, matplotlib 등)
- [ ] AI/ML을 어디에 활용할 것인가? (패턴 분류? 예측? 이상 감지?)
- [ ] 데이터를 어떻게 저장할 것인가? (CSV 파일? 실시간만?)

:::note[대시보드란?]
대시보드는 중요한 정보를 한 화면에 모아서 보여주는 인터페이스예요. 자동차 계기판, 스마트폰 날씨 앱, 주식 차트가 모두 대시보드의 예시입니다. 이번 프로젝트에서는 센서 데이터를 OLED 화면과 PC 화면에 보기 좋게 표시하는 대시보드를 만들어요.
:::

### Step 3. 바이브 코딩 전략 수립 (10분)

프로젝트에서 어떤 부분을 직접 코딩하고, 어떤 부분을 바이브 코딩(AI 도움)으로 해결할지 전략을 세워요.

**바이브 코딩 전략 표:**

| 작업 | 직접 코딩 | 바이브 코딩 | 이유 |
|------|-----------|------------|------|
| 센서 읽기 기본 코드 | O | | 이미 배운 내용이라 직접 할 수 있어요 |
| OLED 대시보드 레이아웃 | | O | 화면 배치 코드는 AI가 더 빠르게 만들어요 |
| 데이터 저장 | O | | CSV 저장은 익숙해요 |
| 웹 대시보드 HTML/JS | | O | 아직 배우지 않은 영역이에요 |
| ML 데이터 수집 코드 | O와 O 혼합 | | 기본 구조는 직접, 세부는 AI 도움 |

:::tip[바이브 코딩 프롬프트 예시]
"Pico WH에서 DHT20 온습도 센서와 소리 센서를 동시에 읽어서, SSD1306 OLED에 실시간 대시보드로 표시하는 MicroPython 코드를 작성해줘. 온도는 큰 글씨, 습도는 막대그래프, 소음 레벨은 경고 아이콘으로 보여줘."
:::

### 도전 코드: 계획서 검증기 + 복잡도 추정

프로젝트 계획서의 완성도와 난이도를 자동으로 평가하는 코드예요.

```python
def validate_and_score(plan):
    """프로젝트 계획서를 검증하고 복잡도를 추정합니다"""

    # 1. 필수 항목 검사
    required = ["team_name", "project_title", "problem",
                "sensors", "dashboard", "ai_ml", "team_roles"]
    missing = [f for f in required if f not in plan or not plan[f]]

    # 2. 복잡도 점수 계산
    complexity = 0

    # 센서 개수에 따른 복잡도
    num_sensors = len(plan.get("sensors", []))
    complexity += num_sensors * 2

    # 대시보드 종류
    dashboard = plan.get("dashboard", {})
    if dashboard.get("oled"):
        complexity += 3
    if dashboard.get("pc"):
        complexity += 4

    # AI/ML 활용
    ai_ml = plan.get("ai_ml", {})
    if ai_ml.get("method"):
        complexity += 5

    # 바이브 코딩 활용도
    vibe = plan.get("vibe_coding", {})
    ai_tasks = len(vibe.get("ai_assisted", []))
    if ai_tasks > 0:
        complexity -= ai_tasks  # 바이브 코딩으로 복잡도 감소

    # 3. 결과 출력
    print("=== 프로젝트 계획서 검증 ===\n")

    if missing:
        print(f"빠진 항목: {', '.join(missing)}")
    else:
        print("필수 항목: 모두 작성됨")

    print(f"\n복잡도 점수: {complexity}점")
    if complexity <= 8:
        level = "적절 (3시간 안에 완성 가능)"
    elif complexity <= 14:
        level = "도전적 (바이브 코딩 적극 활용 권장)"
    else:
        level = "매우 높음 (범위를 줄이거나 역할 분담을 강화하세요)"
    print(f"난이도 평가: {level}")

    # 4. 역할 균형 검사
    roles = plan.get("team_roles", {})
    print(f"\n역할 분담: {len(roles)}명")
    if len(roles) < 3:
        print("  -> 역할이 부족합니다. 최소 3명의 역할을 정하세요.")
    elif len(roles) > 5:
        print("  -> 역할이 많습니다. 핵심 역할로 통합해보세요.")
    else:
        print("  -> 적절한 역할 분담입니다.")

# 우리 팀 계획서로 테스트
validate_and_score(project_plan)
```

:::note[코드 읽어보기]
`plan.get("sensors", [])`는 딕셔너리에서 "sensors" 키를 찾되, 없으면 빈 리스트 `[]`를 반환해요. 이렇게 하면 키가 없어도 에러가 나지 않아요. 복잡도 점수에서 바이브 코딩 항목이 많을수록 점수가 낮아지는 것은, AI 도움으로 개발 부담이 줄어든다는 의미예요.
:::

## 잠깐, 방금 무슨 일이 일어난 걸까요?

여러분이 방금 한 일을 다시 생각해보세요.

"교실이 답답하다"라는 막연한 느낌이
**->** "센서로 측정하고 대시보드로 보여주면 해결할 수 있을까?"라는 **구체적인 질문**이 되고
**->** 센서, 대시보드, AI/ML 활용 방법이 정해지고
**->** 직접 코딩과 바이브 코딩의 **전략적 분담**까지 계획되고
**->** Python 딕셔너리로 **구조화된 프로젝트 계획서**가 만들어졌어요.

이것이 바로 **엔지니어의 사고방식**이에요. 문제를 발견하고, 해결 방법을 설계하고, 도구(센서+코드+AI)를 전략적으로 조합하는 것. 스마트홈, 스마트시티, IoT 시스템 -- 모두 이런 설계 과정에서 시작됩니다.

### OLED 대시보드 스케치

종이에 128x64 픽셀 크기의 사각형을 그리고, 센서 데이터를 어떻게 배치할지 스케치해보세요.

```
+---------------------------+
|  온도: 24.5°C   습도: 62% |
|  ████████░░  CO2: 823ppm  |
|  [쾌적]  밝기: ★★★★☆      |
+---------------------------+
```

### ML 활용 계획

1. **어떤 데이터**를 수집할 것인가요? (예: 5분 간격 온도+습도+CO2)
2. **어떤 카테고리**로 분류할 것인가요? (예: 쾌적/주의/환기필요)
3. **어떤 도구**를 사용할 것인가요? (Teachable Machine? 규칙 기반?)
4. **학습 데이터**는 몇 개 정도 필요할 것 같나요?

:::note[Teachable Machine 미리보기]
13차시에서 Google Teachable Machine을 사용해서 실제 ML 모델을 만들어볼 거예요. 지금은 "어떤 데이터를 모아야 할까?"를 생각해두는 것만으로 충분해요.
